# Open Beverly — Claude Code Instructions

## Project

Static civic transparency site for Beverly, MA. Plain HTML/CSS/JS, no build step.
Deployed to GitHub Pages: https://alaina13.github.io/beverlymadata/ (repo: alaina13/beverlymadata)
Local preview: `python3 -m http.server 8765 --directory /Users/alainapitt/beverlydata` on port 8765.

## Stack

- HTML/CSS/JS only — no framework, no build
- Chart.js 4.4.0 for charts
- MapLibre GL JS 3 for property map
- Font Awesome Kit `56a48e0c30` (script tag in every page)
- `styles.css` — shared design system; all CSS goes here, never inline styles or per-page style blocks
- `nav.js` — shared dropdown nav loaded by every page

## Key pages

- `index.html` — homepage
- `history.html` — fiscal history, Chart.js charts, footnotes
- `whats-next.html` — options, City Hall Q&A, land use, Rogers Building
- `calculator.html` — override calculator
- `property.html` — MapLibre parcel map (13K parcels, GeoJSON)
- `capital.html` — capital plan
- `data.html`, `budget.html`, `meetings.html`, `agendas.html`, `summaries.html` — supporting pages
- `sources.html` — full bibliography

## Key data files

- `parcels.json` — FY2026 assessment data (2.2MB, 13K parcels)
- `parcels_geo.geojson` — parcel polygons with fiscal properties including `ls` (lot size in acres)
- `summaries.json` — AI-generated meeting summaries (generated by `summarize.py`)

## Writing rules (non-negotiable)

- **No em dashes.** Ever. Use a colon, comma, or rewrite. This is a hard rule.
- Neutral, factual tone. No editorializing, no advocacy, no unattributed policy conclusions.
- Every quantitative claim must be sourced or labeled as an estimate.
- No relative dates ("last year") — use absolute dates ("December 2025").
- No "significant" as a vague intensifier. Use numbers or omit.
- Let readers draw their own conclusions.

## Copy workflow

**Always draft copy in conversation first. Wait for explicit approval before committing.**
Applies to: page copy, headings, callout boxes, footnotes, hero text, stat chips, footer text.
Do not push prose changes unilaterally.

## CSS rules

- All CSS goes in `styles.css`. No inline styles, no per-page `<style>` blocks.
- Extend existing shared styles before adding new rules.
- Check for an existing class before writing a new one.

## Git workflow

- After every push, check GitHub Actions status and alert if failed.
- Commit messages: `fix:`, `rewrite:`, `feat:`, `chore:` prefixes.
- `.claude/` is gitignored — do not track it.
